# RFC-0001: Verifiable Convergence Protocol (VCP) v0.1

## 1. Abstract

Verifiable Convergence Protocol (VCP) defines the minimal data types and APIs required for autonomous agents and human-machine systems to achieve epistemic convergence without centralized truth arbiters.

VCP is transport-agnostic, blockchain-agnostic, and model-agnostic. Any system implementing VCP is an Epistemic Execution System (EES).

The key invariant: `C(manipulation) > C(validation)` for all Claim objects.

## 2. Conformance

A system is `VCP-Compatible` iff:
1. It implements the 4 Core Endpoints in Section 4.
2. `confidence` for all Claims is computed via Multiplicative Weights Update + Gossip Consensus as defined in Section 5 (GQRVP).
3. All `Action` objects reference `Claim[]` as `basis`. Actions without basis MUST be rejected at Gateway.
4. `Trace` is immutable, content-addressed, and publicly auditable.

## 3. Core Types

### 3.1. Claim
The atomic unit of verifiable information.
```json
{
  "claim_id": "sha256:...",
  "content": "string",
  "provenance": [{"type": "url|attestation", "src": "string", "signed_by": "did:..."}],
  "method": "string",
  "confidence": "float 0.0-1.0",
  "created_at": "RFC3339 timestamp",
  "expires": "RFC3339 timestamp",
  "trace": ["claim_id"]
}
```

### 3.2. Attestation
Cryptographic signature over a Claim by a DID.
```json
{
  "attestation_id": "sha256:...",
  "claim_id": "sha256:...",
  "issuer": "did:...",
  "signature": "bytes"
}
```

### 3.3. Action
A proposed state change that MUST be grounded in Claims.
```json
{
  "action_id": "sha256:...",
  "name": "string",
  "params": {},
  "basis": ["claim_id"],
  "stake": "uint",
  "proposed_by": "did:..."
}
```

### 3.4. Trace
Immutable record of verification.
```json
{
  "trace_id": "sha256:...",
  "action_id": "sha256:...",
  "result": "ACCEPTED|REJECTED",
  "reason": "string",
  "final_confidence": "float",
  "gqrvp_rounds": "uint",
  "timestamp": "RFC3339 timestamp",
  "merkle_root": "bytes"
}
```

## 4. Core Endpoints

All endpoints MUST be served over HTTPS. Request/response bodies are `application/json`.

### 4.1. POST /v1/claim
Submit a new Claim for verification.
**Request:** `Claim` object without `claim_id`, `confidence`.
**Response 201:** `{"claim_id": "sha256:..."}`
**Response 400:** Invalid Claim schema.

### 4.2. GET /v1/claim/{claim_id}
Retrieve a Claim and its current network confidence.
**Response 200:** `{"claim": Claim, "confidence": 0.92, "attestations": Attestation[], "merkle_proof": "bytes"}`
**Response 404:** Claim not found.

### 4.3. POST /v1/action/verify
Verify if an Action is permitted given its basis Claims.
**Request:** `Action` object.
**Response 200:** `{"status": "ACCEPTED"|"REJECTED", "trace_id": "sha256:...", "reason": "string"}`
Gateway MUST reject if `basis` is empty or any `claim_id` has `confidence < 0.7`.

### 4.4. GET /v1/trace/{trace_id}
Retrieve immutable verification history for an Action.
**Response 200:** `Trace` object.
**Response 404:** Trace not found.

## 5. GQRVP: Security Parameters

To prevent Sybil and amplification attacks, VCP-Compatible implementations MUST use the following parameters for the Grover Amplification Loop:

| Parameter | Symbol | Value | Purpose |
| --- | --- | --- | --- |
| Learning Rate | `η` | `0.2` | Controls weight update speed. Prevents flip-flopping. |
| Amplification | `γ` | `1.5` | Exponent for global score. Amplifies consensus, dampens noise. |
| Mixing | `ε` | `0.1` | Gossip mix ratio. Prevents Sybil cluster isolation. |

A formal proof of 67% Byzantine resilience under these parameters is given in `/specs/RFC/GQRVP-security.md`.

## 7. Security Considerations
VCP does not resolve conflicts caused by asymmetry of goals. VCP guarantees that the cost of manipulating a Claim's `confidence` exceeds the cost of validating it, given an honest supermajority of stake-weighted attention.
