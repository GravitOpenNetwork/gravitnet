1.  Abstract & Introduction

1.1.  Abstract

   Verifiable Convergence Protocol (VCP) defines the minimal data
   types and APIs required for autonomous agents and human-machine
   systems to achieve epistemic convergence without centralized truth
   arbiters. VCP is transport-agnostic, blockchain-agnostic, and
   model-agnostic. Any system implementing VCP is an Epistemic
   Execution System (EES).

   The key invariant is that the cost of manipulating a Claim's
   confidence exceeds the cost of validating it: C(manipulation) >
   C(validation) for all Claim objects.

1.2.  Introduction

   As artificial intelligence transitions to autonomous agents capable
   of independent actions, the lack of a standardized mechanism for
   verifiable trust and epistemic convergence has become a critical
   bottleneck. VCP addresses this by providing a minimal protocol for
   claims, actions, and verifiable traces.

2.  Terminology

   The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT",
   "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in
   this document are to be interpreted as described in RFC 2119.

   - Claim: Atomic unit of verifiable information.
   - Action: Proposed state change grounded in Claims.
   - Trace: Immutable record of verification.
   - Discovery Document: JSON document returned by /.well-known/vcp for
     auto-discovery.
   - EES: Epistemic Execution System.
   - GQRVP: Gravit Quantum Resilient Verification Protocol.

3.  Core Types

3.1.  Claim

   The atomic unit of verifiable information.

   {
     "claim_id": "sha256:...",
     "content": "string",
     "provenance": [
       {
         "type": "url|attestation",
         "src": "string",
         "signed_by": "did:..."
       }
     ],
     "method": "string",
     "confidence": "float 0.0-1.0",
     "created_at": "RFC3339 timestamp",
     "expires": "RFC3339 timestamp",
     "trace": ["claim_id"]
   }

3.2.  Attestation

   Cryptographic signature over a Claim by a DID.

   {
     "attestation_id": "sha256:...",
     "claim_id": "sha256:...",
     "issuer": "did:...",
     "signature": "bytes"
   }

3.3.  Action

   A proposed state change that MUST be grounded in Claims.

   {
     "action_id": "sha256:...",
     "name": "string",
     "params": {},
     "basis": ["claim_id"],
     "stake": "uint",
     "proposed_by": "did:..."
   }

3.4.  Trace

   Immutable record of verification.

   {
     "trace_id": "sha256:...",
     "action_id": "sha256:...",
     "result": "ACCEPTED|REJECTED",
     "reason": "string",
     "final_confidence": "float",
     "gqrvp_rounds": "uint",
     "timestamp": "RFC3339 timestamp",
     "merkle_root": "bytes"
   }

4.  Core Endpoints

   All endpoints MUST be served over HTTPS. Request and response bodies
   are application/json.

4.1.  POST /v1/claim

   Submit a new Claim for verification.
   * Request: Claim object without claim_id, confidence.
   * Response 201: {"claim_id": "sha256:..."}
   * Response 400: {"error": "Invalid Claim schema"}

4.2.  GET /v1/claim/{claim_id}

   Retrieve a Claim and its current network confidence.
   * Response 200:
     {
       "claim": Claim,
       "confidence": 0.92,
       "attestations": Attestation[],
       "merkle_proof": "bytes"
     }
   * Response 404: {"error": "Claim not found"}

4.3.  POST /v1/action/verify

   Verify if an Action is permitted given its basis Claims.
   * Request: Action object.
   * Response 200:
     {
       "status": "ACCEPTED|REJECTED",
       "trace_id": "sha256:...",
       "reason": "string"
     }
   * Rule: Gateway MUST reject if basis is empty or any claim_id has
     confidence < 0.7.

4.4.  GET /v1/trace/{trace_id}

   Retrieve immutable verification history for an Action.
   * Response 200: Trace object.
   * Response 404: {"error": "Trace not found"}

4.5.  GET /.well-known/vcp

   Purpose: Returns the Discovery Document allowing clients to
   automatically discover capabilities, version, and endpoint locations
   of a VCP-compatible node.

   Successful Response (200 OK):
   {
     "subject": "https://node.example.com",
     "vcp_version": "0.1",
     "supported_methods": ["claim", "action/verify", "trace"],
     "endpoints": {
       "claim": "https://node.example.com/v1/claim",
       "claim_get": "https://node.example.com/v1/claim/{claim_id}",
       "action_verify": "https://node.example.com/v1/action/verify",
       "trace": "https://node.example.com/v1/trace/{trace_id}"
     },
     "gqrvp_parameters": {
       "learning_rate": 0.2,
       "amplification": 1.5,
       "mixing": 0.1,
       "threshold": 0.7
     },
     "links": [
       {
         "rel": "https://gravit.network/vcp/schema/claim",
         "href": "https://node.example.com/schemas/claim.json",
         "type": "application/schema+json"
       }
     ]
   }

5.  Protocol Flow

   1. Agent submits Claim -> system computes confidence via GQRVP.
   2. Agent proposes Action with basis Claims.
   3. System verifies and returns Trace.

6.  Security Considerations

   VCP guarantees that the cost of manipulating a Claim's confidence
   exceeds the cost of validating it.

   Discovery Endpoint:
   - Implementations SHOULD apply rate limiting (e.g., 10-30 requests
     per minute per IP).
   - Authentication is OPTIONAL for public nodes but RECOMMENDED for
     private ones.
   - The document MUST NOT expose sensitive data.

   General:
   - All endpoints MUST use HTTPS.
   - Quantum readiness via PQC where applicable.

9.  Implementation Status (per RFC 7942)

   The reference implementation (gravitnet) provides:
   - Implemented: VCP endpoints, in-memory state, discovery via
     /.well-known/vcp.
   - Not implemented: GQRVP MWU+Gossip convergence, signature
     verification, cross-node state synchronization.

   Testnet: A 50-node local cluster can be started using
   scripts/start_testnet.sh.
   - Nodes run in separate containers on a single host (or GitHub Actions
     runner).
   - Each node maintains its own isolated in-memory state.
   - This demonstrates API-level interoperability, not a production
     distributed consensus network.
   - Full gossip-based synchronization is planned for version -01.

   This testnet is a development and demonstration tool, not a production
   deployment.
