Abstract

   This document specifies the Verifiable Convergence Protocol (VCP),
   a minimal, transport-agnostic standard for submitting, verifying,
   and tracing claims in distributed epistemic systems.

   VCP defines four core endpoints (POST /claim, GET /claim/{id},
   POST /action/verify, GET /trace/{id}) and the data types Claim,
   Action, and Trace. It does not define consensus mechanisms,
   tokenomics, or reputation systems. Any system implementing VCP
   is an Epistemic Execution System (EES).

1.  Introduction

   In the emerging ecosystem of autonomous software agents, achieving
   shared agreement over factual claims is a key challenge. Distributed
   agents must establish epistemic convergence without relying on single,
   centralized authorities or resorting to heavy ledger-based consensus.

   The Verifiable Convergence Protocol (VCP) establishes a minimal,
   transport-agnostic standard for submitting, verifying, and tracing
   claims in decentralized systems. By representing claims, attestations,
   and traces as content-addressed schemas, VCP enables autonomous
   systems to evaluate confidence dynamically and audit decisions.

2.  Terminology & Conformance

   The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT",
   "SHOULD", "SHOULD NOT", "RECOMMENDED", "NOT RECOMMENDED", "MAY",
   and "OPTIONAL" in this document are to be interpreted as described
   in BCP 14 [RFC2119] [RFC8174] when, and only when, they appear in
   all capitals, as shown here.

   An Epistemic Execution System (EES) is VCP-Compatible if it:
   1. Implements all core endpoints specified in Section 4.
   2. Computes the network confidence of Claim objects using the GQRVP
      update logic specified in Section 5.
   3. Rejects Action proposals whose basis Claims do not meet the
      minimum verification threshold (confidence >= 0.7).
   4. Generates publicly auditable, content-addressed Trace records.

3.  Core Data Types

3.1.  Claim

   The atomic unit of verifiable information.

   {
     "claim_id": "sha256:...",
     "content": "string",
     "provenance": [
       {
         "type": "url|attestation",
         "src": "string",
         "signed_by": "did:..."
       }
     ],
     "method": "string",
     "confidence": "float",
     "created_at": "RFC3339 timestamp",
     "expires": "RFC3339 timestamp",
     "trace": ["claim_id"]
   }

3.2.  Attestation

   A cryptographic signature over a Claim by a DID.

   {
     "attestation_id": "sha256:...",
     "claim_id": "sha256:...",
     "issuer": "did:...",
     "signature": "bytes"
   }

3.3.  Action

   A proposed state change that MUST be grounded in basis Claims.

   {
     "action_id": "sha256:...",
     "name": "string",
     "params": {},
     "basis": ["claim_id"],
     "stake": "uint",
     "proposed_by": "did:..."
   }

3.4.  Trace

   An immutable record of all validation results applied to an Action.

   {
     "trace_id": "sha256:...",
     "action_id": "sha256:...",
     "result": "ACCEPTED|REJECTED",
     "reason": "string",
     "final_confidence": "float",
     "gqrvp_rounds": "uint",
     "timestamp": "RFC3339 timestamp",
     "merkle_root": "bytes"
   }

4.  Core Endpoints

   All endpoints MUST be served over HTTPS. Requests and responses MUST
   use the "application/json" media type.

4.1.  POST /v1/claim

   Submits a new Claim for verification.
   * Request: A Claim object omitting claim_id and confidence.
   * Response 201: {"claim_id": "sha256:..."}
   * Response 400: {"error": "Invalid Claim schema"}

4.2.  GET /v1/claim/{claim_id}

   Retrieves a Claim and its current network confidence.
   * Response 200:
     {
       "claim": Claim,
       "confidence": float,
       "attestations": Attestation[],
       "merkle_proof": "bytes"
     }
   * Response 404: {"error": "Claim not found"}

4.3.  POST /v1/action/verify

   Verifies whether a proposed Action is permitted given its basis.
   * Request: An Action object.
   * Response 200:
     {
       "status": "ACCEPTED|REJECTED",
       "trace_id": "sha256:...",
       "reason": "string"
     }
   * Rule: The gateway MUST reject the Action if basis is empty, if any
     basis claim_id does not exist, or if any basis Claim has a
     confidence < 0.7.

4.4.  GET /v1/trace/{trace_id}

   Retrieves the immutable verification history for an Action.
   * Response 200: A Trace object.
   * Response 404: {"error": "Trace not found"}

5.  GQRVP & Mathematical Specifications

   The Gravit Quantum Resilient Verification Protocol (GQRVP) models
   agent consensus as a game of Multiplicative Weights Update (MWU)
   with Grover Amplification.

5.1.  Protocol Constants

   VCP-Compatible implementations MUST adhere to the following:
   * Learning Rate (eta): 0.2 (Controls speed of updates; prevents
     flip-flopping).
   * Amplification Exponent (gamma): 1.5 (Exaggerates consensus signal,
     dampening random noise).
   * Mixing Ratio (eps): 0.1 (Prevents cluster isolation in gossip).

5.2.  Confidence Update Algorithm (MWU)

   def update_node_weights(nodes, outcome_losses, eta=0.2):
       """
       Update weights of consensus nodes based on performance.
       """
       total_weight = 0.0
       for node in nodes:
           loss = outcome_losses.get(node.id, 0.0)
           node.weight = node.weight * (1.0 - eta * loss)
           total_weight += node.weight
       if total_weight > 0:
           for node in nodes:
               node.weight /= total_weight

5.3.  Quantum-Resilient Grover Step

   def compute_amplified_consensus(node_votes, weights, gamma=1.5):
       """
       Apply amplification to consensus weights.
       """
       positive_weight = 0.0
       total_weight = 0.0
       for node_id, vote in node_votes.items():
           weight = weights.get(node_id, 0.0)
           total_weight += weight
           if vote == "VALID":
               positive_weight += weight
       if total_weight == 0:
           return 0.0
       ratio = positive_weight / total_weight
       amplified = ((ratio ** gamma) /
                    ((ratio ** gamma) + ((1.0 - ratio) ** gamma)))
       return amplified

6.  Examples (Full Epistemic Flow)

   Step 1: Submit Claim to Node 1
   Request: POST /v1/claim
   {
     "content": "Oracle transaction 0xab92 is valid",
     "provenance": [{"type": "url", "src": "https://..."}],
     "method": "grover",
     "expires": "2026-12-31T23:59:59Z"
   }
   Response 201 Created: {"claim_id": "sha256:d8a9e2f1..."}

   Step 2: Propose Action based on the Claim
   Request: POST /v1/action/verify
   {
     "name": "release_escrow",
     "params": {"recipient": "did:...", "amount": 500},
     "basis": ["sha256:d8a9e2f1..."],
     "stake": 100,
     "proposed_by": "did:..."
   }
   Response 200 OK:
   {
     "status": "ACCEPTED",
     "trace_id": "sha256:ee041b3a...",
     "reason": "All basis claims successfully verified with confidence >= 0.7"
   }

   Step 3: Retrieve Validation Trace
   Request: GET /v1/trace/sha256:ee041b3a...
   Response 200 OK:
   {
     "trace_id": "sha256:ee041b3a...",
     "action_id": "sha256:f12a99d4...",
     "result": "ACCEPTED",
     "reason": "All basis claims successfully verified with confidence >= 0.7",
     "final_confidence": 0.85,
     "gqrvp_rounds": 14,
     "timestamp": "2026-07-17T13:45:00Z",
     "merkle_root": "59a3ee02..."
   }

7.  Interoperability & Reference Implementations

   VCP-Compatible systems rely on two distinct reference implementations:
   * graviteron: An agent runtime host implementing local state stores
     and evaluating GQRVP consensus updates.
   * verifier: An auxiliary microservice that checks Attestation
     signatures and produces auditable cryptographic Trace records.

8.  Privacy & Scalability Considerations

8.1.  Off-Chain Data Storage

   To support scalability and satisfy data privacy compliance (e.g.,
   GDPR), Claim content and provenance metadata SHOULD be stored in
   decentralized content-addressed networks (IPFS, Arweave), while the
   VCP Claim structure retains only the cryptographic hashes.

8.2.  Epistemic Sharding

   Rather than computing MWU updates globally across all EES nodes,
   networks SHOULD partition convergence tasks into domain-specific
   sharding structures. Each shard evaluates sub-networks of related
   Claims, minimizing global message overhead.

8.3.  Zero-Knowledge Attestations

   Future extensions of VCP plan to allow nodes to submit anonymous
   signatures using Zero-Knowledge Proofs (ZKPs) to assert validation
   status without revealing the voting identity or specific stake
   weights of individual nodes.

9.  Security Considerations

   VCP establishes that the cost of manipulating a Claim's confidence
   exceeds the reward, provided that the network's stake-weighted
   attention maintains an honest supermajority.

   Without authentication, endpoints /v1/claim and /v1/action/verify
   are vulnerable to spam and Sybil manipulation if exposed directly
   to the public internet. VCP-Compatible gateways MUST protect these
   routes with transport layer filters or rate limiters.

12.  Implementation Status (per RFC 7942)

   The reference implementation (gravitnet) provides:
   - Implemented: VCP endpoints, in-memory state, discovery via
     /.well-known/vcp.
   - Not implemented: GQRVP MWU+Gossip convergence, signature
     verification, cross-node state synchronization.

   Testnet: A 50-node local cluster can be started using
   scripts/start_testnet.sh.
   - Nodes run in separate containers on a single host (or GitHub Actions
     runner).
   - Each node maintains its own isolated in-memory state.
   - This demonstrates API-level interoperability, not a production
     distributed consensus network.
   - Full gossip-based synchronization is planned for version -01.

   This testnet is a development and demonstration tool, not a production
   deployment.
