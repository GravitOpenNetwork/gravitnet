             Verifiable Convergence Protocol (VCP)
                    draft-gravit-vcp-00.txt

Abstract

   This document specifies the Verifiable Convergence Protocol (VCP),
   a minimal, transport-agnostic standard for submitting, verifying,
   and tracing claims in distributed epistemic systems.

   VCP defines four core endpoints (POST /claim, GET /claim/{id},
   POST /action/verify, GET /trace/{id}) and the data types Claim,
   Action, and Trace. It does not define consensus mechanisms,
   tokenomics, or reputation systems. Any system implementing VCP
   is an Epistemic Execution System (EES).

   The protocol is designed to provide Byzantine resilience against
   Sybil attacks using the GQRVP (Gravit Quantum Resilient Verification
   Protocol) parameters: eta=0.2, gamma=1.5, epsilon=0.1. These ensure
   that the cost of manipulating a Claim's confidence exceeds the cost
   of validating it, given an honest supermajority of stake-weighted
   attention.

1.  Introduction

   In the emerging ecosystem of autonomous agents (AI, bots, oracles),
   a fundamental problem persists: how can agents agree on a shared
   reality without relying on a central authority? Existing solutions
   are either too heavy (blockchain consensus), too weak (reputation
   systems), or too specific (oracle networks).

   The Verifiable Convergence Protocol (VCP) defines a minimal
   interoperability standard for Epistemic Execution Systems (EES) -
   systems where agents can submit Claims, verify them through
   cryptographic Attestations and Actions, and converge on ground
   truth via stake-weighted attention.

   VCP is not a blockchain. It does not define a token, a consensus
   algorithm, or a network topology. It defines only the data types
   and API endpoints necessary for agents to coordinate on the
   veracity of information.

   The protocol's security is grounded in the GQRVP (Gravit Quantum
   Resilient Verification Protocol) model, which provides Byzantine
   resilience against Sybil attacks using Multiplicative Weights
   Update, Grover Amplification, and Gossip Consensus.

2.  Terminology

   The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT",
   "SHOULD", "SHOULD NOT", "RECOMMENDED", "NOT RECOMMENDED", "MAY",
   and "OPTIONAL" in this document are to be interpreted as described
   in BCP 14 [RFC2119] [RFC8174] when, and only when, they appear in
   all capitals, as shown here.

   This document uses the following terms:

   o  Claim: A structured assertion about a state of the world,
      submitted for verification.

   o  Attestation: A cryptographic signature over a Claim by a DID.

   o  Action: A proposed state change that MUST be grounded in one or
      more Claims.

   o  Trace: An immutable record of all Verification Actions applied
      to a Claim.

   o  EES (Epistemic Execution System): Any system that implements
      VCP endpoints and adheres to the Security Considerations.

   o  Confidence: A floating-point value between 0.0 and 1.0
      representing the network's belief in a Claim's veracity.

3.  Core Data Types

3.1.  Claim

   The atomic unit of verifiable information.

   {
     "claim_id": "sha256:...",
     "content": "string",
     "provenance": [
       {"type": "url|attestation", "src": "string", "signed_by": "did:..."}
     ],
     "method": "string",
     "confidence": "float 0.0-1.0",
     "created_at": "RFC3339 timestamp",
     "expires": "RFC3339 timestamp",
     "trace": ["claim_id"]
   }

   The "provenance" field MUST contain at least one entry.

   The "method" field indicates the verification method used
   (e.g., "grover", "human", "consensus", "oracle").

3.2.  Attestation

   A cryptographic signature over a Claim by a DID.

   {
     "attestation_id": "sha256:...",
     "claim_id": "sha256:...",
     "issuer": "did:...",
     "signature": "bytes"
   }

3.3.  Action

   A proposed state change that MUST be grounded in Claims.

   {
     "action_id": "sha256:...",
     "name": "string",
     "params": {},
     "basis": ["claim_id"],
     "stake": "uint",
     "proposed_by": "did:..."
   }

   The "basis" field MUST contain at least one Claim ID.
   Actions without basis MUST be rejected at the Gateway.

3.4.  Trace

   Immutable record of verification.

   {
     "trace_id": "sha256:...",
     "action_id": "sha256:...",
     "result": "ACCEPTED|REJECTED",
     "reason": "string",
     "final_confidence": "float",
     "gqrvp_rounds": "uint",
     "timestamp": "RFC3339 timestamp",
     "merkle_root": "bytes"
   }

4.  Core Endpoints

   All endpoints MUST be served over HTTPS.
   Request and response bodies are "application/json".

4.1.  POST /v1/claim

   Submit a new Claim for verification.

   Request: Claim object without "claim_id", "confidence".

   Response 201:
   {"claim_id": "sha256:..."}

   Response 400:
   {"error": "Invalid Claim schema"}

4.2.  GET /v1/claim/{claim_id}

   Retrieve a Claim and its current network confidence.

   Response 200:
   {
     "claim": Claim,
     "confidence": 0.92,
     "attestations": Attestation[],
     "merkle_proof": "bytes"
   }

   Response 404:
   {"error": "Claim not found"}

4.3.  POST /v1/action/verify

   Verify if an Action is permitted given its basis Claims.

   Request: Action object.

   Response 200:
   {
     "status": "ACCEPTED|REJECTED",
     "trace_id": "sha256:...",
     "reason": "string"
   }

   Gateway MUST reject if "basis" is empty or any "claim_id" has
   "confidence < 0.7".

4.4.  GET /v1/trace/{trace_id}

   Retrieve immutable verification history for an Action.

   Response 200: Trace object.
   Response 404: {"error": "Trace not found"}

5.  GQRVP Security Parameters

   To prevent Sybil and amplification attacks, VCP-Compatible
   implementations MUST use the following parameters for the
   Grover Amplification Loop:

   +----------+--------+-------+--------------------------------------+
   | Parameter| Symbol | Value | Purpose                              |
   +----------+--------+-------+--------------------------------------+
   | Learning | eta    | 0.2   | Controls weight update speed.        |
   | Rate     |        |       | Prevents flip-flopping.              |
   | Amplifi- | gamma  | 1.5   | Exponent for global score.           |
   | cation   |        |       | Amplifies consensus, dampens noise.  |
   | Mixing   | eps    | 0.1   | Gossip mix ratio. Prevents Sybil     |
   |          |        |       | cluster isolation.                   |
   +----------+--------+-------+--------------------------------------+

   A formal proof of 33% Byzantine resilience under these parameters
   is given in [GQRVP-SECURITY].

6.  Security Considerations

   VCP does not resolve conflicts caused by asymmetry of goals.
   VCP guarantees that the cost of manipulating a Claim's "confidence"
   exceeds the cost of validating it, given an honest supermajority
   of stake-weighted attention.

   The GQRVP security model assumes that honest nodes have a
   higher initial weight than Sybils. This is enforced by
   Proof-of-Attestation in the Trust Engine.

   For a full discussion of attack vectors and mitigations,
   see [GQRVP-SECURITY].
